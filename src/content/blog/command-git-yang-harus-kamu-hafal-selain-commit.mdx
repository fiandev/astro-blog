## Kenyataan Pahit Developer Pemula Tentang Git

Kalau kamu jujur sama diri sendiri, kemungkinan besar workflow Git kamu sehari hari masih muter di pola yang sama.

> Edit kode, lalu **git add**, **git commit**, dan terakhir **git push**. Selesai.

Di tahap awal belajar coding, itu sudah cukup. Bahkan bagus. Tapi masalahnya, **banyak developer berhenti di situ terlalu lama**. Mereka bisa bikin fitur, bisa deploy, tapi panik saat:

*   Salah commit
*   Perlu ambil satu perubahan kecil dari branch lain
*   Harus membatalkan commit di production
*   Ditanya histori perubahan oleh senior atau lead

Di titik ini, Git bukan lagi alat bantu, tapi sumber stres.

Padahal Git itu bukan cuma alat simpan kode. Git adalah **alat komunikasi antar developer**. Cara kamu pakai Git mencerminkan cara kamu berpikir sebagai engineer.

Artikel ini bakal ngebahas command Git penting selain commit yang **wajib kamu hafal kalau mau naik level**, khususnya:

*   git stash
*   git cherry pick
*   git revert
*   git log

Bukan sekadar apa itu, tapi **kapan dipakai, kenapa penting, dan dampaknya ke karier kamu**.

## Kenapa Banyak Developer Salah Kaprah Soal Git

Salah satu kesalahan paling umum di generasi developer muda adalah menganggap Git itu cuma syarat teknis.

Yang penting bisa push ke repository, selesai.

Padahal di dunia kerja profesional:

*   Kode jarang dikerjakan sendirian
*   Kesalahan kecil bisa berdampak ke banyak orang
*   Histori perubahan itu krusial buat audit dan debugging

Banyak developer:

*   Copy paste command tanpa ngerti efeknya
*   Takut nyentuh Git karena takut rusak
*   Menghindari konflik alih alih memahaminya

Akibatnya:

*   Workflow berantakan
*   Tim susah kolaborasi
*   Developer terlihat tidak dewasa secara teknis

Menguasai Git di luar commit bukan soal sok jago. Ini soal **bertanggung jawab terhadap kode yang kamu sentuh**.

## Git Stash

### Apa Itu Git Stash

Git stash adalah fitur untuk **menyimpan perubahan sementara tanpa harus commit**.

Bayangin kamu lagi ngerjain sesuatu, tapi tiba tiba:

*   Disuruh pindah branch
*   Harus pull update dari remote
*   Ada bug urgent yang harus ditangani

Kode kamu belum siap di commit, tapi juga nggak boleh hilang.  
 Di sinilah git stash jadi penyelamat.

Secara sederhana, git stash itu seperti:

> “Simpan dulu kerjaanku, nanti balikin lagi.”

### Kapan Git Stash Dipakai

Git stash sering dipakai di situasi nyata seperti:

*   Lagi ngoding fitur, tapi ada hotfix mendadak
*   Mau pull branch terbaru tapi working tree kotor
*   Lagi eksperimen tapi belum yakin hasilnya

Daripada:

*   Commit setengah matang
*   Copy paste kode ke file lain
*   Panik dan bingung

Git stash memberi solusi bersih dan profesional.

### Command Git Stash yang Wajib Dihafal

Beberapa command dasar yang wajib kamu kuasai:

*   **git stash**  
    Menyimpan perubahan ke stash
*   **git stash list**  
    Melihat daftar stash yang pernah disimpan
*   **git stash apply**  
    Mengambil kembali stash tanpa menghapusnya dari daftar
*   **git stash pop**  
    Mengambil stash sekaligus menghapusnya dari daftar
*   **git stash drop**  
    Menghapus stash tertentu

Kalau kamu sering kerja di banyak task, stash ini bakal sering dipakai.

### Kesalahan Umum Saat Pakai Git Stash

Banyak developer pemula melakukan kesalahan seperti:

*   Lupa pernah stash
*   Stash numpuk dan tidak pernah dibersihkan
*   Apply stash di branch yang salah

Best practice sederhana:

*   Selalu cek **git stash list**
*   Beri konteks sebelum apply
*   Jangan anggap stash sebagai penyimpanan jangka panjang

Git stash itu alat bantu sementara, bukan gudang.

## Git Cherry Pick

### Apa Itu Git Cherry Pick

Git cherry pick digunakan untuk **mengambil satu commit spesifik dari branch lain** dan menerapkannya ke branch saat ini.

Bukan merge. Bukan rebase.  
 Tapi benar benar memilih satu commit saja.

Ini sangat berguna saat:

*   Kamu hanya butuh satu bugfix kecil
*   Tidak ingin membawa seluruh perubahan branch lain

### Skenario Nyata Penggunaan Cherry Pick

Contoh kasus nyata di dunia kerja:

*   Ada bugfix penting di branch development
*   Branch production butuh fix itu sekarang
*   Tapi fitur lain di development belum siap

Daripada merge seluruh branch dan bikin masalah baru, kamu cukup cherry pick commit bugfix tersebut.

Ini workflow yang sering dipakai di tim profesional.

### Cara Menggunakan Git Cherry Pick dengan Aman

Langkah umumnya:

1.  Cari hash commit dengan git log
2.  Jalankan **git cherry-pick <hash>**

Kalau terjadi konflik:

*   Selesaikan konflik
*   Lanjutkan cherry pick

Kalau ternyata salah:

*   Cherry pick bisa dibatalkan

Cherry pick itu powerful, tapi harus digunakan dengan sadar.

### Dampak Cherry Pick ke Histori Git

Hal penting yang harus kamu tahu:

*   Cherry pick menciptakan commit baru
*   Histori bisa jadi duplikat
*   Tidak cocok untuk workflow jangka panjang

Cherry pick sebaiknya dipakai untuk:

*   Fix cepat
*   Kasus spesifik
*   Bukan kebiasaan harian

Mengerti batasannya itu tanda developer dewasa.

## Git Revert

### Apa Itu Git Revert

Git revert digunakan untuk **membatalkan efek sebuah commit dengan cara aman**.

Bukan menghapus histori, tapi membuat commit baru yang isinya kebalikan dari commit sebelumnya.

Ini sangat penting untuk:

*   Branch publik
*   Repository tim
*   Production environment

### Bedanya Git Revert dan Git Reset

Ini kesalahan klasik yang sering bikin masalah besar.

*   Git reset mengubah histori
*   Git revert menambah histori

Di branch yang sudah dipakai banyak orang, reset bisa bikin chaos.  
 Revert adalah pilihan aman dan profesional.

### Kenapa Git Revert Penting di Project Tim

Di dunia kerja:

*   Semua perubahan harus bisa dilacak
*   Kesalahan harus terlihat
*   Audit itu penting

Git revert memastikan:

*   Histori tetap utuh
*   Tidak merusak kerja orang lain
*   Aman untuk kolaborasi

Ini alasan kenapa revert sering dipakai di production.

### Cara Menggunakan Git Revert

Beberapa skenario umum:

*   Revert satu commit
*   Revert beberapa commit
*   Revert merge commit

Semua itu bisa dilakukan tanpa menghapus histori.

Kalau kamu belum pernah revert di production, berarti kamu belum cukup lama di dunia nyata.

### Kesalahan Fatal Saat Revert

Kesalahan yang sering terjadi:

*   Revert commit yang salah
*   Tidak cek git log sebelum revert
*   Panik dan revert berulang kali

Pelajaran penting:

> Jangan buru buru. Git selalu punya jejak.

## Git Log

### Kenapa Git Log Sering Diremehkan

Banyak developer jarang pakai git log.  
 Padahal ini salah satu command paling penting.

Git log itu:

*   Mata kamu ke histori project
*   Alat belajar dari kode orang lain
*   Senjata debugging yang kuat

### Git Log untuk Membaca Alur Project

Dengan git log, kamu bisa:

*   Tahu siapa mengubah apa
*   Kapan perubahan dilakukan
*   Kenapa sebuah fitur muncul

Ini penting banget saat:

*   Masuk ke project baru
*   Onboarding di tim
*   Maintain codebase lama

### Variasi Git Log yang Wajib Dicoba

Beberapa kombinasi berguna:

*   Tampilan ringkas
*   Visual graph branch
*   Filter berdasarkan author
*   Filter berdasarkan waktu

Git log itu fleksibel dan sangat powerful.

### Git Log sebagai Alat Belajar Developer Muda

Kalau kamu mau belajar dari developer senior:

*   Buka repository open source
*   Pakai git log
*   Baca commit message
*   Pelajari alur berpikir mereka

Ini cara belajar yang jarang diajarkan, tapi sangat efektif.

## Kombinasi Command Git yang Bikin Kamu Terlihat Pro

Developer yang terlihat matang biasanya:

*   Stash sebelum pull
*   Log sebelum revert
*   Cherry pick dengan alasan jelas
*   Tidak asal reset

Bukan soal hafal command, tapi **tahu kapan dipakai**.

## Dampak Menguasai Git ke Personal Branding

Menguasai Git di luar commit memberi dampak besar ke citra kamu sebagai developer:

*   Terlihat paham workflow tim
*   Lebih dipercaya menangani branch penting

*   Dianggap siap kerja, bukan cuma bisa ngoding

Di dunia profesional, **cara kamu pakai Git sering lebih diperhatikan daripada syntax yang kamu hafal**.
